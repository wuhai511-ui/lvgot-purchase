# 旅购通代码审查报告

> 审查日期：2026-04-04  
> 项目：whf-fund-deposit-demo  
> 审查范围：server/ (Node.js/Express API) + js/ (前端)

---

## 总体评价

**综合评分：5.5 / 10**

项目整体具备可运行的资金预存基本框架，架构设计清晰，API 结构合理。但存在多个**安全性 P0 问题**（JWT 密钥硬编码、无权限控制、敏感路由开放）、**业务逻辑缺陷**（退款不减 totalSpent、状态机从未使用 PENDING 状态）、**数据模型不规范**（无 Prisma 关联关系、无外键约束）、**测试覆盖率极低**，以及多处重复代码和性能隐患。项目目前处于"演示可用、上线危险"的状态。

## 问题清单（按优先级）

### 🔴 P0 - 必须修复（上线阻断）

#### 1. JWT 密钥硬编码，密码强度极弱
**文件：** `server/routes/auth.js` 第 6 行
```javascript
const JWT_SECRET = process.env.JWT_SECRET || 'whf-fund-deposit-secret-key-2026';
```
**风险：** 如果环境变量未设置，默认密钥被写死在代码中，任何能访问源码的人均可伪造任意用户 JWT。
**修复：**
```javascript
const JWT_SECRET = process.env.JWT_SECRET;
if (!JWT_SECRET) {
  throw new Error('JWT_SECRET environment variable is required');
}
```

#### 2. 所有 API 路由完全无权限控制
**文件：** `server/routes/transactions.js` `server/routes/ledgers.js` `server/routes/merchants.js` 等所有路由文件
**问题：** 除了 `/auth/*` 以外的路由（如 `/transactions/deposit`、`/merchants`）没有任何中间件验证请求者身份和权限，任何人可随意操作用户资金。
**修复：** 引入认证中间件，所有业务路由强制 JWT 验证：
```javascript
// middleware/auth.js
function requireAuth(req, res, next) {
  const auth = req.headers.authorization;
  if (!auth?.startsWith('Bearer ')) {
    return res.status(401).json({ code: 401, message: '未授权' });
  }
  try {
    req.user = jwt.verify(auth.slice(7), JWT_SECRET);
    next();
  } catch {
    res.status(401).json({ code: 401, message: 'Token无效' });
  }
}
```

#### 3. 注册接口对公网开放
**文件：** `server/routes/auth.js` 第 47 行
```javascript
router.post('/register', (req, res) => { ... }
```
**风险：** 生产环境中任何人均可注册管理员账户。
**修复：** 仅在 `NODE_ENV === 'development'` 时开放注册路由，或完全移除注册 API。

#### 4. 退款（refund）不减 totalSpent，导致账目不平衡
**文件：** `server/routes/transactions.js` 第 142-168 行
```javascript
// refund 路由中：
prisma.ledger.update({
  where: { id: ledger.id },
  data: { balance: balanceAfter, updatedAt: new Date() }  // ❌ 没有更新 totalSpent
}),
```
**后果：** 假设用户存入 1000 元，消费 300 元（totalSpent=300，balance=700），退款 200 元后 balance=900，但 totalSpent 仍为 300，则 `totalDeposited(1000) ≠ totalSpent(300) + balance(900)`，账目不等。
**修复：**
```javascript
data: {
  balance: balanceAfter,
  totalSpent: Number(ledger.totalSpent) - Number(amount),  // 退款应减少已消费
  updatedAt: new Date()
}
```

#### 5. Prisma Schema 缺少关联关系定义（@relation）
**文件：** `server/prisma/schema.prisma`
```prisma
model Ledger {
  merchant_id      String  // ❌ 无 @relation，无 references
  consumer_id     String
}
model Transaction {
  ledger_id       String  // ❌ 同上
  merchant_id     String
  consumer_id     String
}
```
**后果：** Prisma 无法验证外键完整性，无法通过 `ledger.merchant`、`tx.ledger()` 等关联查询；级联删除也无法配置。
**修复：** 添加完整 @relation：
```prisma
model Ledger {
  id           String @id @default(cuid())
  merchantId   String
  merchant     Merchant @relation(fields: [merchantId], references: [id])
  consumerId   String
  consumer     Consumer @relation(fields: [consumerId], references: [id])
  transactions Transaction[]
  @@unique([merchantId, consumerId])  // 防止重复 Ledger
}
```

#### 6. 资金字段使用 Float 而非 Decimal
**文件：** `server/prisma/schema.prisma` — 所有 amount、balance 等字段
```prisma
balance        Float   // ❌
amount         Float   // ❌
```
**后果：** JavaScript Float 计算会产生精度问题（如 `0.1 + 0.2 = 0.30000000000000004`），资金计算可能产生微小偏差累积。
**修复：** 使用 Prisma Decimal 类型（底层映射为 SQL NUMERIC/DECIMAL）：
```prisma
balance        Decimal
amount         Decimal
```

### 🟠 P1 - 建议修复（上线前应处理）

#### 7. deposit 接口不验证商户/消费者是否存在
**文件：** `server/routes/transactions.js` 第 46-73 行
```javascript
// deposit 路由直接创建 Ledger，不检查 merchant/consumer 是否有效
let ledger = await prisma.ledger.findFirst({ where: { merchantId, consumerId } });
if (!ledger) {
  ledger = await prisma.ledger.create({ data: { merchantId, consumerId, ... } });
}
```
**后果：** 可能创建孤立 Ledger（指向不存在的商户或消费者）。
**修复：** 先查询验证商户和消费者存在性，再创建 Ledger。

#### 8. 支付渠道（QZT）调用无重试机制
**文件：** `server/core/payment/qzt-client.js` 第 68-93 行 `doRequest()` 函数
```javascript
const response = await fetch(QZT_CONFIG.serviceUrl, {
  ...
  signal: AbortSignal.timeout(QZT_CONFIG.timeout),
});
```
**后果：** 网络瞬时故障（如超时、503）直接返回失败，资金状态不确定（可能扣款但系统未记录）。
**修复：** 实现 3 次指数退避重试：
```javascript
async function doRequestWithRetry(requestBody, retries = 3) {
  for (let i = 0; i < retries; i++) {
    try {
      return await doRequest(requestBody);
    } catch (err) {
      if (i === retries - 1) throw err;
      await new Promise(r => setTimeout(r, 1000 * Math.pow(2, i)));
    }
  }
}
```

#### 9. withdraw 接口无幂等性保护
**文件：** `server/routes/transactions.js` 第 76-115 行
**问题：** 用户可重复提交同一笔提款请求（网络重试等场景），导致重复扣款。
**修复：** 使用幂等键（Idempotency Key）机制，在 Transaction 表加唯一索引 `(ledger_id, idem_key)`，或利用 `outRequestNo` 防止重复。

#### 10. 初始化脚本密码硬编码
**文件：** `server/db/init.js` 第 50 行
```javascript
const hashedPwd = bcrypt.hashSync('admin123', 10);
```
**风险：** 演示密码泄露，且固定 salt 导致彩虹表攻击可行。
**修复：** 从环境变量读取默认密码，或强制首次登录后修改。

#### 11. 错误处理返回详细信息给客户端
**文件：** 多处 `res.status(500).json({ code: 500, message: err.message })`
**风险：** `err.message` 可能包含数据库结构、文件路径等敏感信息。
**修复：** 日志记录详细信息，对外返回通用错误信息：
```javascript
console.error('预存失败:', err);  // 写日志
res.status(500).json({ code: 500, message: '服务器内部错误' });  // 对外隐藏细节
```

#### 12. CORS 配置允许所有来源
**文件：** `server/index.js` 第 13 行
```javascript
app.use(cors());  // 无参数 = 允许所有来源
```
**修复：** 生产环境应指定白名单：
```javascript
app.use(cors({
  origin: process.env.ALLOWED_ORIGINS?.split(',') || ['http://localhost:3000']
}));
```

#### 13. Prisma 连接未使用连接池管理
**文件：** 所有路由文件
```javascript
const prisma = new PrismaClient({ adapter });
// 每请求调用 prisma.$disconnect() — 错误用法
```
**问题：** `prisma.$disconnect()` 每请求关闭连接是**错误用法**，PrismaClient 设计为长连接，自动维护连接池。
**修复：** 移除所有 `finally { await prisma.$disconnect(); }`，PrismaClient 在进程结束时自动断开。

#### 14. 交易状态机不完整（pending/failed 从未被使用）
**文件：** `server/routes/transactions.js` 所有交易路由
**问题：** 所有交易立即创建为 `status: 'completed'`，pending 和 failed 状态从未被使用。无法实现"支付中→成功/失败"的状态流转。
**修复：** deposit/withdraw 应先创建 `pending` 交易，支付回调后更新为 `completed` 或 `failed`。

#### 15. 敏感密钥文件存放在代码仓库
**文件：** `server/core/payment/qzt-client.js` 第 10-11 行
```javascript
const KEYS_DIR = path.resolve(__dirname, '../../tests/keys');
```
**风险：** 私钥/公钥放在 tests/ 目录并提交到 git 仓库。
**修复：** 密钥应通过环境变量指定路径，或使用云密钥管理服务（AWS KMS / 阿里云 KMS）。

### 🟡 P2 - 优化建议

#### 16. 大量重复代码 — Prisma 实例初始化
**位置：** 每个路由文件（transactions.js、ledgers.js、merchants.js 等）均复制以下代码：
```javascript
const dbPath = path.join(__dirname, '..', 'prisma', 'dev.db');
const adapter = new PrismaLibSql({ url: `file:${dbPath}` });
const prisma = new PrismaClient({ adapter });
```
**修复：** 抽取到 `server/db/prisma.js` 统一导出：
```javascript
// server/db/prisma.js
const adapter = new PrismaLibSql({ url: `file:${path.join(__dirname, '../prisma/dev.db')}` });
module.exports = new PrismaClient({ adapter });
```

#### 17. 响应格式转换重复
**位置：** 每个路由都有类似代码：
```javascript
const result = transactions.map(t => ({
  ...t,
  amount: Number(t.amount),
  balance_before: Number(t.balanceBefore),
  ...
}));
```
**修复：** 使用 Prisma `$queryRaw` + 直接别名，或实现统一响应转换中间件。

#### 18. Prisma Schema 命名不一致
**问题：** Schema 中混合使用 snake_case（`merchant_id`、`consumer_id`、`created_at`）和 camelCase（`updatedAt`）。
**Prisma 规范：** 推荐全部使用 camelCase，数据库层用 `@map("merchant_id")` 映射。
```prisma
model Ledger {
  merchantId   String @map("merchant_id")
  createdAt    DateTime @default(now()) @map("created_at")
}
```

#### 19. 统计接口性能问题
**文件：** `server/routes/stats.js` 第 11-32 行 `overview` 路由
```javascript
const [merchants, consumers, ledgers, transactions, pendingTx, pendingSettlements] =
  await Promise.all([...六次全表查询...]);
```
**修复：** 合并为 1-2 条聚合查询，使用 Prisma `groupBy` 或原生 SQL。

#### 20. /transactions 列表无最大限制
**文件：** `server/routes/transactions.js` 第 12 行
```javascript
take: parseInt(pageSize)  // pageSize=50 默认，可被绕过
```
**风险：** 大数据量下仍可请求 pageSize=10000 导致内存问题。
**修复：** 设置上限：
```javascript
take: Math.min(parseInt(pageSize), 100)
```

#### 21. 初始化数据脚本无幂等性
**文件：** `server/db/init.js`
**问题：** 每次运行会重复插入演示数据（虽然有 check，但多次运行可能）。
**修复：** 使用 `prisma.$executeRaw` 或迁移脚本。

#### 22. 前端 XSS 风险
**文件：** `js/app.js` 第 144-153 行 `renderMerchants` 等函数
```javascript
opt.text = typeof m === 'string' ? m : `${m.name} (${m.id})`;
```
**问题：** 直接将后端返回的商户名插入 DOM，未做 HTML 转义。
**修复：** 对所有用户输入内容使用 `textContent` 而非 `innerHTML`。

#### 23. 路由文件无统一前缀验证中间件
**问题：** 所有路由手动解析 JWT，分散且容易遗漏。
**修复：** 在 `index.js` 中全局注册 auth 中间件，或为各路由组注册中间件。

#### 24. refund 金额未做上限检查
**文件：** `server/routes/transactions.js` 第 142 行
**问题：** 退款金额可超过实际消费金额（超额退款）。
**修复：** 校验 `amount <= totalSpent - totalRefunded`。

#### 25. 缺少输入参数校验中间件
**问题：** 每个路由手动检查必填字段，重复且易遗漏。
**修复：** 引入 `joi` 或 `zod` 进行 schema 校验。

## 亮点（做得好的地方）

1. **架构设计清晰**：分层明确（路由 → 业务逻辑 → 数据访问），ARCHITECTURE.md 文档详尽
2. **双重数据库支持**：同时维护 JSON 文件数据库（无依赖演示用）和 Prisma/SQLite（生产用），过渡平滑
3. **事务原子性**：`deposit`、`withdraw`、`refund` 均使用 `prisma.$transaction` 保证账务一致性
4. **幂等 ID 生成**：使用 UUID 生成交易 ID，避免 ID 冲突
5. **Mock 降级方案**：前端支持 API 不可用时自动降级到 Mock 数据，保证演示可用性
6. **分页设计**：主要列表接口均实现了 page + pageSize 分页
7. **前后端分离**：API 设计符合 REST 规范，前端通过统一 API 模块封装请求
8. **QZT 支付 SDK 完整性**：签名验签、文件上传、OCR、身份证认证等接口封装完整
9. **类型转换规范**：统一将 Prisma Decimal/Float 转为明确数字类型
10. **测试文件结构**：qzt.test.js 的单元测试结构可复用

### 业务逻辑

| 流程 | 状态 | 说明 |
|------|------|------|
| 资金预存（deposit） | ⚠️ 部分正确 | 不验证商户/消费者存在，无幂等 |
| 消费扣款（withdraw） | ✅ 基本正确 | 有余额检查，使用事务 |

3. 退款 200（bug）→ totalSpent 仍为 300, balance=900
4. **验证失败：** totalDeposited(1000) ≠ totalSpent(300) + balance(900=1200)

### 数据模型

**Prisma Schema 问题汇总：**

| 问题 | 影响 | 文件 |
|------|------|------|
| 无 @relation 定义 | 无法关联查询，外键不约束 | `schema.prisma` |
| Ledger 无 @@unique([merchantId, consumerId]) | 同一用户在同一商户可创建多个 Ledger | `schema.prisma` |
| 货币字段用 Float | 精度丢失风险（金融大忌） | `schema.prisma` 所有 amount/balance |
| tags 存为 String | 无法数组操作，且未映射 | `schema.prisma` |
| 无 @@index on status/createdAt | 常用过滤条件无索引 | `schema.prisma` |
| Merchant.balance 字段冗余 | balance 应只存在于 Ledger（账自分离） | `schema.prisma` |
| Consumer.open_id/phone 可为 null 但 unique | 可能产生多条无 openid 记录 | `schema.prisma` |
| Settlement.net_amount 未自动计算 | 需要手动维护一致性 | `schema.prisma` |

**Ledger 与 Transaction 的 balance 字段是冗余设计** — Transaction 已经记录了每笔的 balanceBefore/balanceAfter，Ledger.balance 是这些值的投影。缺少任何一方更新则不一致（参考上述 refund bug）。

### 性能

|------|---------|------|
| 每请求 disconnect Prisma | 🟠 浪费连接 | 所有路由 `finally` 块 |
| /merchants/:id N+1 查询 | 🟠 | `merchants.js GET /:id` |
| /stats/overview 6 次全表查询 | 🟡 | `stats.js overview` |
| /stats/transactions 内存过滤 | 🟡 大数据量问题 | `stats.js transactions` |
| transaction 列表无上限 | 🟡 | `transactions.js` |
| ledgers/:id include transactions 无分页 | 🟡 | `ledgers.js GET /:id` |

**最严重性能问题：**
```javascript
// ledgers.js GET /:id — 包含所有历史交易无分页
include: { transactions: { orderBy: { createdAt: 'desc' } } }
// 如果一个 Ledger 有 10 万条交易，响应将极大
```
