## 一、前言

### 1.1 需求基本信息

| 项目 | 说明 |
|------|------|
| **需求名称** | 旅购通分账通道升级改造 |
| **需求类型** | 系统升级 / 通道迁移 |
| **核心目标** | 将原有的**备付金分账模式**全面升级为**基于拉卡拉支付账户的分账模式**，突破资金占用和分账层级的双重限制。 |
| **涉及系统** | 小冲系统、钱账通、钱包系统 |
| **涉及平台** | 旅购通 App、钱账通运营后台 |

### 1.2 名词解释

| 术语 | 定义 |
|------|------|
| **支付账户** | 基于拉卡拉支付体系开立的数字账户，支持资金收付、余利增值及分账功能。是本次升级的核心载体。 |
| **银行内部户** | 挂接在银行底层的内部虚拟账户。在开立支付账户时可选择性开启，用于特定银行结算逻辑映射。新模式下默认关闭。 |
| **结算补贴** | 针对特定银行卡（如指定分行卡）进行提现或手续费的政策性补贴，通过在后台配置特定绑卡规则实现。用于引导用户绑卡、优化财务结算成本。 |
| **备付金模式** | 旧版分账模式，资金沉淀在备付金账户，存在最低留存比例限制，分账层级受限。 |
| **分账** | 将一笔交易金额按规则分配给多个收款方的操作，支持全额余额分账（无留存）。 |
| **活体检测** | 通过人脸识别确认账户持有人身份，是支付账户激活的必要条件。 |
| **钱包系统** | 钱账通底层支付账户能力提供方，负责账户开户、审核、活体认证等核心能力。 |
| **小冲系统** | 旅购通业务前端系统，负责用户交互、订单创建、分账发起等业务逻辑。 |

## 二、需求背景

### 2.1 要解决的核心问题

#### 问题一：取消最低留存比例限制

- **现状：** 原备付金模式要求账户必须保留一定比例的余额（最低留存），导致资金周转率低，商户资金被无效占用。
- **目标：** 新模式支持**全额分账/提现**，账户余额可全部支配，资金利用率提升至 100%。

#### 问题二：突破分账层级限制

- **现状：** 当前不支持多级分账，无法满足旅游场景中"商户 → 旅行社 → 导游/司机"的链条式分账需求。
- **目标：** 实现**多级链条分账**，支持任意两级（含）以上的分账关系。

#### 问题三：个人入驻流程合规与体验矛盾

- **现状：** 原模式对导游/司机（个人用户）要求"强人脸识别"方可入驻，转化流失率高。
- **目标：** 改为"**入驻不强制、升级强校验**"模式，降低初始转化门槛，同时在涉及大额交易时强制要求账户升级（活体认证）。

### 2.2 预期效果

| 维度 | 现状（旧） | 目标（新） |
|------|-----------|-----------|
| 分账模式 | 备付金分账 | 拉卡拉支付账户分账 |
| 资金留存 | 最低留存比例强制保留 | 全额分账/提现，零留存 |
| 分账层级 | 单级或受限 | 多级链条（商户→旅行社→导游/司机） |
| 个人入驻 | 强人脸识别强制 | 先入驻，升级时强校验 |
| 账户体系 | 单一账户 | 支付账户 + 可选银行内部户（按配置） |
| 财务成本 | 固定费率 | 可配置结算补贴，优化成本 |

## 三、用户角色与账户体系

### 3.1 角色定义

| 角色 | 类型 | 典型用户 | 核心诉求 |
|------|------|----------|----------|
| **商户** | 企业/对公 | 旅游商品零售商店、免税店 | 收款、查看交易流水、余额提现 |
| **旅行社** | 企业/对公 | 旅行社总社/分社 | 向导游/司机分账、接收商店分账、资金归集 |
| **导游/司机** | 个人 | 导游、司机 | 接收分账、余额提现、日常结算 |
| **运营管理员** | 系统角色 | 钱账通后台运营 | 审核商户资质、配置应用参数、查看交易数据 |

### 3.2 账户体系说明

#### 3.2.1 支付账户（核心）

每个角色在钱账通开立一个**支付账户**，具有以下特性：

- **账户状态：** `INVALID`（无效）/ `VALID`（有效）
- **余额类型：** 可用余额、冻结余额
- **分账权限：** 账户状态为 VALID 时方可参与分账（作为付款方或收款方）
- **提现权限：** 账户状态为 VALID 时可发起提现

#### 3.2.2 银行内部户（可选，按应用配置）

- **开通方式：** 非强制，通过钱账通后台应用配置开关控制（默认关闭）
- **适用场景：** 需要特定银行结算逻辑映射的商户
- **配置路径：** 钱账通运营后台 → 应用管理 → 编辑应用配置 → 开通银行内部户（是/否）

#### 3.2.3 角色分账关系图

```
┌──────────────────────────────────────────────────────┐
│                    分账流向示意                         │
│                                                      │
│   ┌────────┐     分账收款      ┌──────────┐          │
│   │  商户   │ ──────────────→ │  旅行社   │          │
│   │(旅游商店)│                 │          │          │
│   └────────┘                 └────┬─────┘          │
│        ↑                              │             │
│        │ 分账收款                     │ 分账收款    │
│        │                              ▼             │
│   收付款/提现              ┌──────────────────┐     │
│                             │   导游 / 司机    │     │
│                             │    (个人账户)    │     │
│                             └──────────────────┘     │
└──────────────────────────────────────────────────────┘
```

**说明：**
- 商户（旅游商店）可向旅行社分账
- 旅行社可向导游/司机分账
- 旅行社也可接收来自商户的分账
- 所有分账参与方账户状态必须为 VALID

## 五、账户状态机

### 5.1 状态定义

| 状态 | 代码 | 说明 |
|------|------|------|
| **无效** | `INVALID` | 账户已开立但未完成激活，不可进行分账/提现 |
| **有效** | `VALID` | 账户已完成所有激活流程，可正常进行所有资金操作 |

### 5.2 状态流转规则

#### 5.2.1 对公账户（企业/门店）流程

```
[提交资料]
     │
     ▼
[审核_待审核] ──(资质不符)──→ [审核_驳回] ──→ 用户修改资料重新提交
     │
 (钱包系统审核通过)
     │
     ▼
[审核_已通过] ──────────────→ [对公活体_待检测]
                                        │
                                 (完成人脸识别)
                                        │
                                        ▼
                                  [活体_通过]
                                        │
                                        ▼
                                  [开户_有效] ✅
```

**触发条件：**
- `审核_待审核 → 审核_已通过`：钱包系统人工/自动审核通过
- `审核_待审核 → 审核_驳回`：资质不符合要求
- `审核_已通过 → 对公活体_待检测`：审核通过后自动进入
- `对公活体_待检测 → 活体_通过`：法人完成人脸识别
- `活体_通过 → 开户_有效`：最终激活，账户可用

#### 5.2.2 个人账户（导游/司机）流程

```
[个人申请]
     │
     ▼
[个人活体_待检测] ───(无需审核，自动通过)
     │
(完成人脸识别)
     │
     ▼
[活体_通过]
     │
     ▼
[开户_有效] ✅
```

**触发条件：**
- `个人申请 → 个人活体_待检测`：提交即跳过审核环节，直接进入活体
- `个人活体_待检测 → 活体_通过`：用户完成人脸识别
- `活体_通过 → 开户_有效`：识别成功，账户立即激活

### 5.3 状态关联逻辑规则表

| 账户类型 | 审核状态 | 活体状态 | 最终开户状态 | 业务行为说明 |
|---------|---------|---------|------------|------------|
| 对公账户 | 待审核 | 待检测 | **无效** | 资料已提交，等待运营人工/系统审核 |
| 对公账户 | 通过 | 待检测 | **无效** | 资质已通过，需法人完成人脸识别以激活账户 |
| 对公账户 | 通过 | 通过 | **有效** ✅ | 账户完全激活，可收款、分账、提现 |
| 个人账户 | 不涉及（自动通过） | 待检测 | **无效** | 提交即进入活体环节，人脸识别前不可用 |
| 个人账户 | 不涉及（自动通过） | 通过 | **有效** ✅ | 人脸识别完成，账户立即激活 |

### 5.4 状态驱动行为规则

| 当前状态 | 发起分账 | 发起提现 | 发起入驻升级 |
|---------|---------|---------|------------|
| INVALID（待审核） | ❌ 拒绝 | ❌ 拒绝 | ❌ 禁止操作 |
| INVALID（待活体） | ❌ 拒绝（引导升级） | ❌ 拒绝（引导升级） | ✅ 允许 |
| VALID | ✅ 仅余额部分 | ✅ 有限额（引导升级提额） | ⚠️ 已升级（不显示入口） |

### 6.1 【钱账通】业务应用配置

#### 6.1.1 需求描述

在钱账通后台的应用配置中，新增两个关键配置项，按应用独立控制：

1. **是否强制开通银行内部户**（默认：否）
2. **是否强制进行电子签约**（默认：否）

#### 6.1.2 用户旅程

```
运营管理员登录后台
    │
    ▼
进入「应用管理」→ 选择目标应用
    │
    ▼
编辑应用配置
    │
    ├── Toggle开关：「开通银行内部户」(是/否)
    │
    └── Toggle开关：「开启电子签约」(是/否)
    │
    ▼
保存配置
```

#### 6.1.3 业务逻辑判断

| 配置项 | 值为"是" | 值为"否"（默认） |
|--------|---------|----------------|
| 开通银行内部户 | 开户流程包含银行内部户开立步骤 | 开户流程跳过银行内部户开立步骤 |
| 开启电子签约 | 开户流程包含电子签约步骤 | 开户流程跳过电子签约步骤 |

**默认行为：**
- 新应用默认为"否"（不强制），与旧版本逻辑兼容
- 若该应用此前已有商户入驻，修改配置不影响已开户账户

#### 6.1.4 输入与输出

### 6.2 【旅购通+钱账通】支付账户开户

```

③ 提交资料
    │

    ▼
④ 立即跳转活体识别页面
⑤ 完成人脸识别
⑥ 识别成功 → 账户激活
⑦ 显示"开户成功"
```

**对公用户（商户/旅行社）：**

#### 6.3.1 需求描述

支持导游/司机个人用户通过"升级账户"路径补办活体认证，解决账户大额资金处理的合规性问题。底层调用钱包系统的人脸识别能力。

#### 6.3.2 用户旅程

```
① 用户在"我的账户"页面点击"升级账户"按钮
    │

    │
    ├── 已升级 → 提示"您已完成升级"，不再显示升级入口
    │
    └── 未升级 → 进入下一步
        │
        ▼
③ 系统调用钱账通接口，获取人脸识别 H5 链接/Token
    │
    ▼
④ 前端唤起人脸识别 H5 页面（或 WebView）
    │
    ▼
⑤ 用户完成动作活体识别
    │

| 判断项 | 逻辑 |

| 重复升级校验 | 校验是否已升级，避免重复操作浪费资源 |
| 鉴权方式 | 人脸识别由钱包系统完成，钱账通仅透传 Token |
| 结果通知 | 支持异步回调（Webhook）和前端轮询（`GET /api/account/upgrade/result`）两种模式 |

| account_id | String | 是 | 支付账户ID |

**输出：**

| 字段 | 类型 | 说明 |
|------|------|------|
| face_auth_token | String | 人脸识别 Token，用于前端跳转 |
| face_auth_url | String | 人脸识别 H5 页面 URL |
