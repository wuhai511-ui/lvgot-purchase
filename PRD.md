## 一、前言

### 1.1 需求基本信息

| 项目 | 说明 |
|------|------|
| **需求名称** | 旅购通分账通道升级改造 |
| **需求类型** | 系统升级 / 通道迁移 |
| **核心目标** | 将原有的**备付金分账模式**全面升级为**基于拉卡拉支付账户的分账模式**，突破资金占用和分账层级的双重限制。 |
| **涉及系统** | 小冲系统、钱账通、钱包系统 |
| **涉及平台** | 旅购通 App、钱账通运营后台 |

### 1.2 名词解释

| 术语 | 定义 |
|------|------|
| **支付账户** | 基于拉卡拉支付体系开立的数字账户，支持资金收付、余利增值及分账功能。是本次升级的核心载体。 |
| **银行内部户** | 挂接在银行底层的内部虚拟账户。在开立支付账户时可选择性开启，用于特定银行结算逻辑映射。新模式下默认关闭。 |
| **结算补贴** | 针对特定银行卡（如指定分行卡）进行提现或手续费的政策性补贴，通过在后台配置特定绑卡规则实现。用于引导用户绑卡、优化财务结算成本。 |
| **备付金模式** | 旧版分账模式，资金沉淀在备付金账户，存在最低留存比例限制，分账层级受限。 |
| **分账** | 将一笔交易金额按规则分配给多个收款方的操作，支持全额余额分账（无留存）。 |
| **活体检测** | 通过人脸识别确认账户持有人身份，是支付账户激活的必要条件。 |
| **钱包系统** | 钱账通底层支付账户能力提供方，负责账户开户、审核、活体认证等核心能力。 |
| **小冲系统** | 旅购通业务前端系统，负责用户交互、订单创建、分账发起等业务逻辑。 |

## 二、需求背景

### 2.1 要解决的核心问题

#### 问题一：取消最低留存比例限制

- **现状：** 原备付金模式要求账户必须保留一定比例的余额（最低留存），导致资金周转率低，商户资金被无效占用。
- **目标：** 新模式支持**全额分账/提现**，账户余额可全部支配，资金利用率提升至 100%。

#### 问题二：突破分账层级限制

- **现状：** 当前不支持多级分账，无法满足旅游场景中"商户 → 旅行社 → 导游/司机"的链条式分账需求。
- **目标：** 实现**多级链条分账**，支持任意两级（含）以上的分账关系。

#### 问题三：个人入驻流程合规与体验矛盾

- **现状：** 原模式对导游/司机（个人用户）要求"强人脸识别"方可入驻，转化流失率高。
- **目标：** 改为"**入驻不强制、升级强校验**"模式，降低初始转化门槛，同时在涉及大额交易时强制要求账户升级（活体认证）。

### 2.2 预期效果

| 维度 | 现状（旧） | 目标（新） |
|------|-----------|-----------|
| 分账模式 | 备付金分账 | 拉卡拉支付账户分账 |
| 资金留存 | 最低留存比例强制保留 | 全额分账/提现，零留存 |
| 分账层级 | 单级或受限 | 多级链条（商户→旅行社→导游/司机） |
| 个人入驻 | 强人脸识别强制 | 先入驻，升级时强校验 |
| 账户体系 | 单一账户 | 支付账户 + 可选银行内部户（按配置） |
| 财务成本 | 固定费率 | 可配置结算补贴，优化成本 |

## 三、用户角色与账户体系

### 3.1 角色定义

| 角色 | 类型 | 典型用户 | 核心诉求 |
|------|------|----------|----------|
| **商户** | 企业/对公 | 旅游商品零售商店、免税店 | 收款、查看交易流水、余额提现 |
| **旅行社** | 企业/对公 | 旅行社总社/分社 | 向导游/司机分账、接收商店分账、资金归集 |
| **导游/司机** | 个人 | 导游、司机 | 接收分账、余额提现、日常结算 |
| **运营管理员** | 系统角色 | 钱账通后台运营 | 审核商户资质、配置应用参数、查看交易数据 |

### 3.2 账户体系说明

#### 3.2.1 支付账户（核心）

每个角色在钱账通开立一个**支付账户**，具有以下特性：

- **账户状态：** `INVALID`（无效）/ `VALID`（有效）
- **余额类型：** 可用余额、冻结余额
- **分账权限：** 账户状态为 VALID 时方可参与分账（作为付款方或收款方）
- **提现权限：** 账户状态为 VALID 时可发起提现

#### 3.2.2 银行内部户（可选，按应用配置）

- **开通方式：** 非强制，通过钱账通后台应用配置开关控制（默认关闭）
- **适用场景：** 需要特定银行结算逻辑映射的商户
- **配置路径：** 钱账通运营后台 → 应用管理 → 编辑应用配置 → 开通银行内部户（是/否）

```

│                    分账流向示意                         │
│                                                      │
│   ┌────────┐     分账收款      ┌──────────┐          │
│   │  商户   │ ──────────────→ │  旅行社   │          │
│   │(旅游商店)│                 │          │          │
│   └────────┘                 └────┬─────┘          │
│        ↑                              │             │
│        │ 分账收款                     │ 分账收款    │
│        │                              ▼             │
│   收付款/提现              ┌──────────────────┐     │
│                             │   导游 / 司机    │     │

│                             └──────────────────┘     │
```

**说明：**
- 商户（旅游商店）可向旅行社分账
- 旅行社可向导游/司机分账
- 旅行社也可接收来自商户的分账
- 所有分账参与方账户状态必须为 VALID

## 五、账户状态机

### 5.1 状态定义

| 状态 | 代码 | 说明 |
|------|------|------|
| **无效** | `INVALID` | 账户已开立但未完成激活，不可进行分账/提现 |
| **有效** | `VALID` | 账户已完成所有激活流程，可正常进行所有资金操作 |

### 5.2 状态流转规则

#### 5.2.1 对公账户（企业/门店）流程

```
[提交资料]
     │
     ▼
[审核_待审核] ──(资质不符)──→ [审核_驳回] ──→ 用户修改资料重新提交
     │
 (钱包系统审核通过)
     │
     ▼
[审核_已通过] ──────────────→ [对公活体_待检测]
                                        │
                                 (完成人脸识别)
                                        │
                                        ▼
                                  [活体_通过]
                                        │
                                        ▼
                                  [开户_有效] ✅
```

**触发条件：**
- `审核_待审核 → 审核_已通过`：钱包系统人工/自动审核通过
- `审核_待审核 → 审核_驳回`：资质不符合要求
- `审核_已通过 → 对公活体_待检测`：审核通过后自动进入
- `对公活体_待检测 → 活体_通过`：法人完成人脸识别
- `活体_通过 → 开户_有效`：最终激活，账户可用

#### 5.2.2 个人账户（导游/司机）流程

```
[个人申请]
     │
     ▼
[个人活体_待检测] ───(无需审核，自动通过)
     │
(完成人脸识别)
     │
     ▼
[活体_通过]
     │
     ▼
[开户_有效] ✅
```

**触发条件：**
- `个人申请 → 个人活体_待检测`：提交即跳过审核环节，直接进入活体
- `个人活体_待检测 → 活体_通过`：用户完成人脸识别
- `活体_通过 → 开户_有效`：识别成功，账户立即激活

### 5.3 状态关联逻辑规则表

| 账户类型 | 审核状态 | 活体状态 | 最终开户状态 | 业务行为说明 |
|---------|---------|---------|------------|------------|
| 对公账户 | 待审核 | 待检测 | **无效** | 资料已提交，等待运营人工/系统审核 |
| 对公账户 | 通过 | 待检测 | **无效** | 资质已通过，需法人完成人脸识别以激活账户 |
| 对公账户 | 通过 | 通过 | **有效** ✅ | 账户完全激活，可收款、分账、提现 |
| 个人账户 | 不涉及（自动通过） | 待检测 | **无效** | 提交即进入活体环节，人脸识别前不可用 |
| 个人账户 | 不涉及（自动通过） | 通过 | **有效** ✅ | 人脸识别完成，账户立即激活 |

### 5.4 状态驱动行为规则

| 当前状态 | 发起分账 | 发起提现 | 发起入驻升级 |
|---------|---------|---------|------------|
| INVALID（待审核） | ❌ 拒绝 | ❌ 拒绝 | ❌ 禁止操作 |
| INVALID（待活体） | ❌ 拒绝（引导升级） | ❌ 拒绝（引导升级） | ✅ 允许 |
| VALID | ✅ 仅余额部分 | ✅ 有限额（引导升级提额） | ⚠️ 已升级（不显示入口） |

### 6.1 【钱账通】业务应用配置

#### 6.1.1 需求描述

在钱账通后台的应用配置中，新增两个关键配置项，按应用独立控制：

1. **是否强制开通银行内部户**（默认：否）
2. **是否强制进行电子签约**（默认：否）

#### 6.1.2 用户旅程

```
运营管理员登录后台
    │
    ▼
进入「应用管理」→ 选择目标应用
    │
    ▼
编辑应用配置
    │
    ├── Toggle开关：「开通银行内部户」(是/否)
    │
    └── Toggle开关：「开启电子签约」(是/否)
    │
    ▼
保存配置
```

#### 6.1.3 业务逻辑判断

| 配置项 | 值为"是" | 值为"否"（默认） |
|--------|---------|----------------|
| 开通银行内部户 | 开户流程包含银行内部户开立步骤 | 开户流程跳过银行内部户开立步骤 |
| 开启电子签约 | 开户流程包含电子签约步骤 | 开户流程跳过电子签约步骤 |

**默认行为：**
- 新应用默认为"否"（不强制），与旧版本逻辑兼容
- 若该应用此前已有商户入驻，修改配置不影响已开户账户

#### 6.1.4 输入与输出

### 6.10 【钱账通】银行开户和支付账户解耦

#### 6.10.1 需求描述

商户角色不再强制要求开通银行内部户，是否开立银行内部户由应用配置（`enable_bank_internal_account`）控制。

#### 6.10.2 业务逻辑

| 配置值 | 行为 |
|--------|------|
| `true`（开启） | 开户流程包含银行内部户开立步骤 |
| `false`（关闭，默认） | 开户流程跳过银行内部户开立步骤，仅开立支付账户 |

#### 6.10.3 影响范围

| 模块 | 影响说明 |
|------|---------|
| 开户流程 | 根据配置决定是否调用银行内部户开立接口 |
| 提现逻辑 | 银行内部户关闭时，提现直接走支付账户通道 |
| 后台展示 | 账户详情页根据配置决定是否展示银行内部户信息 |

### 6.11 【钱账通】支付账户开户协议可配置

#### 6.11.1 需求描述

商户和分账方是否需要电子签约，由应用配置（`enable_electronic_sign`）控制。

#### 6.11.2 业务逻辑

| 配置值 | 行为 |
|--------|------|
| `true`（开启） | 开户流程包含电子签约步骤 |
| `false`（关闭，默认） | 开户流程跳过电子签约步骤 |

**默认行为：**
- 新应用默认为"否"（不强制电子签约）
- 历史已签约用户不受影响

### 6.12 【钱账通】账户权限控制

#### 6.12.1 需求描述

账户是否可转出（分账/转账），由钱账通后台按应用配置控制。

#### 6.12.2 配置维度

| 配置项 | 说明 |
|--------|------|
| `transfer_out_enabled` | 是否允许转出（分账/转账给他人） |
| `withdraw_enabled` | 是否允许提现至银行卡 |

#### 6.12.3 业务逻辑

| 配置状态 | 分账行为 | 提现行为 |
|---------|---------|---------|
| 允许 | 正常分账 | 正常提现 |
| 禁止 | 返回错误：账户转出权限已被关闭 | 返回错误：账户提现权限已被关闭 |

### 6.13 【钱账通】运营后台菜单调整

#### 6.13.1 需求描述

对钱账通运营后台的菜单结构进行重构，符合新的交易管理逻辑。

#### 6.13.2 具体调整明细

| 序号 | 原位置 | 调整后位置 | 变更类型 |
|------|--------|----------|--------|
| 1 | `账户管理` → 账户方充值转账 | → `交易管理` → 账户方充值转账 | 移动 |
| 2 | `账户管理` → 结算单 | → `交易管理` → **支付账户提现** | 移动+改名 |
| 3 | 无 | `交易管理` → 余额分账 → 增加"导出"按钮 | 新增 |
| 4 | 各交易查询页（无条件） | 各交易查询页（无条件）→ 增加"支付账户号"搜索条件 | 查询增强 |

#### 6.13.3 业务逻辑

- 纯粹的菜单路由重定向和 SQL 查询条件拼接，无复杂业务逻辑
- 移动后的菜单 URL 需做 301 重定向，防止历史书签失效

#### 6.13.4 涉及接口

| 接口 | 变更说明 |
|------|---------|
| 后台管理列表查询接口 | 需新增 `payment_account_no` 入参 |
| 余额分账列表接口 | 需新增 `export` 功能（导出 Excel） |

### 7.2 关键交互说明

#### 7.2.1 动态鉴权组件

分账和提现页面的鉴权方式根据 `payment_auth_type` 配置动态渲染：

| 配置 | 界面行为 |
|------|---------|
| `NONE` | 不显示任何鉴权控件，直接显示"确认提交"按钮 |
| `SMS` | 显示"获取验证码"按钮（倒计时 60s）+ 6位数字输入框 |
| `PWD` | 显示支付密码输入框（6位，点击弹出数字键盘） |

#### 7.2.2 活体识别唤起

账户开户和升级两个场景均需唤起人脸识别：

- **开户场景（对公）：** 审核通过后，App 弹窗/跳转至人脸识别 H5 页面（法人操作）
- **开户场景（个人）：** 资料提交成功后，直接跳转活体识别，无需等待
- **升级场景：** 点击"升级账户"后，获取 Token 并跳转 H5 页面

#### 7.2.3 账户状态引导

| 账户状态 | 用户感知 | 引导动作 |
|---------|---------|---------|
| 待审核 | "资料已提交，请等待审核" | 显示审核进度 |
| 待活体（对公） | "审核已通过，请完成身份认证激活账户" | 高亮"去认证"按钮 |
| 待活体（个人） | "请完成身份认证以激活账户" | 自动跳转活体识别 |
| 有效 | "账户已激活，可使用全部功能" | 正常展示余额和可用操作 |
| 驳回 | "资料审核未通过，请修改后重新提交" | 显示驳回原因+修改入口 |

## 十、风险与依赖

### 10.1 风险清单

| 风险 ID | 风险描述 | 严重程度 | 概率 | 影响 | 应对措施 |
|---------|---------|---------|------|------|---------|
| R-01 | **钱包系统接口变更** — 若拉卡拉钱包系统升级导致接口协议变更，本次改造的对接代码需同步调整 | 高 | 中 | 工期延误 | 提前与拉卡拉对接方确认接口稳定性；接口层做版本兼容 |
| R-02 | **活体识别失败率** — 人脸识别受网络、设备、光线影响，存在一定失败率，可能导致用户开户失败率上升 | 中 | 中 | 用户体验下降 | 提供重试机制；失败超过 3 次后引导联系客服 |
| R-03 | **老用户迁移** — 已开户老用户（备付金模式）的账户余额和分账关系迁移，存在数据一致性风险 | 高 | 低 | 用户资金损失 | 制定详细的灰度迁移方案；迁移前全量数据校验；回滚预案 |
| R-04 | **对公账户审核积压** — 人工审核环节可能因审核人力不足导致对公开户周期过长 | 中 | 中 | 用户投诉增加 | 提前评估审核人力；审核页面增加预计等待时间提示 |
| R-05 | **分账链路断裂** — 链路上任意一环（商户/旅行社/导游）账户状态异常，导致整条分账失败 | 中 | 低 | 部分分账场景无法完成 | 增加前置状态校验，提前告知用户；支持部分成功分账 |
| R-06 | **电子签约合规性** — 电子签约的法律效力认定，需确认符合《电子签名法》要求 | 高 | 低 | 合规风险 | 确认签约平台资质；签约记录需支持出证 |
| R-07 | **结算补贴配置错误** — 后台配置结算补贴规则时出错，导致财务结算成本超出预期 | 中 | 低 | 财务损失 | 配置修改需二次确认；建立配置变更审计日志 |
